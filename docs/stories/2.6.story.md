## Story

**Como** aluno do workshop "Living Lab Azure-Native",
**eu quero** completar a Fase 6 em 8h implementando uma nova rota `/flow` no frontend Vite que mostra, em tempo real via SignalR, uma "bolinha" animada percorrendo APIM → Service Bus → Function Consumer → n8n → SQL para cada compra v2,
**para que** eu compreenda distributed tracing, correlation IDs, observabilidade com App Insights, e termine o workshop com a visualização mais didática possível do que aprendi nas 5 fases anteriores — capaz de demonstrar a um leigo a complexidade desacoplada que existe por trás de "comprar um ingresso".

## Acceptance Criteria

1. **AC-1 — Branch criada:** `phase-06-flow-visualizer` partindo de `phase-05-ai-mcp`, workflow `.github/workflows/deploy-phase-06.yml`.

2. **AC-2 — Azure SignalR Service provisionado via Portal** (free tier, 20 conexões, East US 2): `signalr-fifa2026-<iniciais>`.

3. **AC-3 — Function `FlowEventsFunction`** dedicada: consulta App Insights por correlation ID via SDK e empurra eventos via SignalR.

4. **AC-4 — Correlation ID propagado em TODOS os componentes:**
   - APIM: `set-header X-Correlation-ID` (já em F2)
   - Function Entry: lê header, anexa em ILogger scope
   - Service Bus: incluído nas application properties da mensagem
   - Function Consumer: lê das application properties, anexa em logs
   - n8n: webhook envia correlationId no payload
   - SQL: já tem coluna `correlation_id` (desde F1)
   - W3C Trace Context honored em todos os hops

5. **AC-5 — Rota `/flow` no frontend Vite:**
   - Diagrama dos 5 nós (APIM → Function → SB → Consumer → n8n → SQL)
   - Lista das últimas 50 compras (correlationId, timestamp, status)
   - Busca por correlationId

6. **AC-6 — Animação em tempo real:**
   - Ao clicar numa compra, "bolinha" framer-motion percorre o diagrama
   - Cada nó mostra tempo gasto + status (ok/erro) + payload inspecionável (Sheet/Dialog)
   - SignalR connection (fallback polling 2s se WebSocket falhar)

7. **AC-7 — Smoke test ponta-a-ponta:** fazer compra v2 → ir para `/flow` → ver bolinha percorrer 5 nós em < 30s → cada nó com tempo + payload corretos.

8. **AC-8 — Performance:** página `/flow` carrega em < 1.5s; animação fluida (60fps em Chrome moderno).

9. **AC-9 — Acessibilidade:** rota `/flow` keyboard-navigable; nós com `aria-label`; modo "lista" para users sem JS animation.

10. **AC-10 — Merge final → main:** após DoD de S2.6, abrir PR `phase-06-flow-visualizer` → `main`. Tag `v2.0.0` ao mergear.

## Dev Notes

### Arquitetura técnica delta

```
[Cliente] → /flow?id=<correlationId>
[Vite frontend]
  ├─ useSignalRConnection(correlationId)
  │   └─ conecta SignalR Service via JS SDK
  └─ FlowDiagram <motion.svg>
        └─ recebe eventos via SignalR
              ├─ APIM_RECEIVED (timestamp, duration)
              ├─ FUNCTION_ENTRY_PROCESSED
              ├─ SERVICE_BUS_PUBLISHED
              ├─ FUNCTION_CONSUMER_PROCESSED
              ├─ N8N_WEBHOOK_TRIGGERED
              └─ SQL_INSERTED

[FlowEventsFunction]
  ├─ Query App Insights por correlationId
  ├─ Map traces → eventos tipados
  └─ Push via SignalR para grupo correlation-<id>
```

### Correlation propagation matrix (essencial revisar de cada fase)

| Hop | Como propaga | Lê de | Escreve em |
|---|---|---|---|
| APIM → Function Entry | header X-Correlation-ID | (gerado se ausente) | header + log scope |
| Function Entry → SB | application property | log scope | message.ApplicationProperties["CorrelationId"] |
| SB → Function Consumer | application property | message.ApplicationProperties | log scope |
| Function Consumer → SQL | coluna correlation_id | log scope | purchases.correlation_id |
| Function Consumer → n8n | body do webhook | log scope | payload.correlationId |
| n8n → log | console.log | payload.correlationId | structured log |

### SignalR setup mínimo

- Service Mode: Default (não Serverless — precisa Hub clássico)
- Hub: `FlowHub`
- Method: `SendFlowEvent(correlationId, eventType, timestamp, durationMs, payload)`
- Client subscribe: `connection.on("FlowEvent", handler)`

### App Insights Kusto query exemplo

```kusto
traces
| where customDimensions.CorrelationId == "<correlationId>"
| order by timestamp asc
| project timestamp, message, customDimensions, severityLevel
| limit 100
```

- E2E: Playwright runs purchase → navigate to /flow → assert 5 nodes ativadas
- Performance: Lighthouse CI no GitHub Actions
- Accessibility: axe-core no Playwright
- Visual regression: Percy ou Chromatic (opcional)
