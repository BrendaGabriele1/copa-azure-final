## Story

**Como** aluno do workshop "Living Lab Azure-Native",
**eu quero** completar a Fase 3 em 6h ativando autenticação via Microsoft Entra External ID no fluxo v2 (preservando o JWT+bcrypt do v1 para comparação didática) e configurando App Registration para a camada admin,
**para que** eu compreenda CIAM moderno (OIDC, social login, scopes) e a diferença entre Entra interno (admin) e External (customer), e tenha a policy `validate-jwt` do APIM ativada — preparando o terreno para chatbot e Visualizer.

## Dev Notes

### Arquitetura técnica delta

```
[Browser]
   │ Login no External ID (OIDC)
   │ recebe access_token (Bearer)
   ▼
[APIM Developer]
   │ policy validate-jwt: valida issuer + audience
   │ propaga claims via header X-Entra-OID
   ▼
[Function PurchaseEntry]
   │ lê X-Entra-OID
   │ mapeia para user_id local (conforme ADE-001)
   │ continua fluxo F1
```

### Comparação v1 vs v2 (para README — AC-10)

| Aspecto | v1 (intocado) | v2 (F3) |
|---|---|---|
| Storage de senha | bcrypt(10 rounds) em users.password_hash | Não armazenado (provider externo) |
| Token | JWT HS256 local | JWT RS256 do Entra |
| Issuer | fifa2026-api local | extid-fifa2026.ciamlogin.com |
| Validação | middleware Express manual | APIM policy validate-jwt |
| Social login | Não | Sim (Google/MS) |
| Refresh | Não | Sim (OAuth2) |
| MFA | Não | Sim (configurável) |

### Decisões Arquiteturais Esperadas (ADE-001)

Três opções:
- **A: Tabela mapping** `user_identity_mapping (user_id, entra_oid, created_at)`
- **B: Coluna em users** `users.entra_oid UNIQUEIDENTIFIER NULL`
- **C: Duplicar registro** novo `users.id` para cada Entra OID

`@architect` decide e justifica em ADE-001.

### Testing

**Test approach:**
- Integration: spinear MSAL test harness OU mock do issuer com chave pública conhecida
- Smoke: login real → token → API call → SQL persistido
- Security: token expirado → 401; token de outro tenant → 401
